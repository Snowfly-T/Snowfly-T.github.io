很多项目真的需要分页吗？

很多项目真的需要分页吗？

这是个大多数前端和后端看到了都会觉得荒谬的问题……而且应该说几乎所有后端看到这个问题都会觉得很荒谬，比前端还要稍多点

但巧合的是，我不知为啥正好处于看到这问题不觉得荒谬的少数人之一，为什么呢

因为我自己 Web 是从前端学起的，然后再学的后端。虽然说现在主要写的是后端，但我个人感觉可能还是更熟悉前端一点。

我看到的大多数前端同学都是一开始就习惯于当一个“渲染层”，负责将后端给的数据展示出来，而不做太多处理的（没有任何贬义的成分）。而这也是大多数公司里前端做的活（至少是国内），数据处理几乎全部交给后端，而前端仅作极少的数据变换就展示出来。我觉得这可能也是 GraphQL 在国内火不起来的原因之一……

但是前端能做的事情真挺多的。说来不怕人笑话，我在学校里当时第一个真正拿去参加比赛的作品，表面光鲜，但背后的实现非常搞笑：后端直接提供一个给前端传 SQL 的接口，前端直接写 SQL 传给后端！但是你别说，如果完全不考虑安全性，由于前端可以完全控制想获取什么样的数据，所以效率其实非常高。当然，这多少有点回到前后端不分离的时代了，属实开时代倒车了。

我想说的其实是只要不是计算密集型或是那种超高并发的任务，把数据变换的主要逻辑放到前端真的不会造成什么性能上的问题。我看到有人在这问题底下说前端要搞“分页”需要考虑数据一致性、冗余数据等一堆问题……是的，这些也是问题，但是现在有了 Tanstack Query（旧称 React Query）这样的库之后做起来已经方便多了，不比后端实现来得复杂。这类库就是在前端请求数据这块加个中间层，帮你把缓存、更新等问题统一管理，免得你再去拿 IndexDB 或其他东西手动在前端搓个缓存层。如果不考虑后端一次传太多数据导致的请求超时问题，其实前端来做这些事情单纯在复杂度上并没有增加，毕竟早就有轮子把这些事给做好了。

至于页面卡死问题……各种懒加载的实现方案很多，不要一次把所有数据渲染出来就行……

其实想想看像是表格中数据的排序工作、检索工作这些本来就应该是前端做的事情。按理来说，后端返回原始数据，前端把数据缓存下来，用户检索、排序、筛选时用这个缓存的前端数据，才是最优雅简洁的方案，而且某种意义上，其实也降低了后端的压力——前端一次请求全部数据会造成压力，但是用户频繁搜索、筛选数据其实也会造成后端压力……如果数据量本来就不大，顶多几百条数据，何必后端先做分页，再用 Redis 去解决前边提到的用户反复搜索、筛选相似数据的问题呢

其实相似的，很多数据处理工作前端来做是很合适的，也方便降低后端压力。我个人的习惯就是后端提供相对粗粒度的接口，而一些细节工作交给前端封装一下去完成，这样其实也是为了降低后端负载。其实这种想法并不少见啊，再深入一些，就是中间层的那套思路，拿 Node 搭个中间层其实原理是类似的，也是把一些细节工作放到另一层来减少后端负载。另一个层面上，GraphQL 也是这个思路，前端细粒度地告诉后端需要什么数据，而后端本身提供的只是一大堆粗粒度的数据，然后根据前端的需求只响应必要的数据。

想想为什么一个请求响应速度慢，不就三个层面吗：

  • 数据库读写太多
  • 逻辑复杂，后端计算量大
  • 网络传输速度慢

这三个原因从主要到次要，“网络传输速度慢”只是最次要的原因，只有当前两个层面都解决的很好的时候，性能问题才可能出现在这一点上——当然，真要遇上这种情况，那肯定只能分页了。

但关键是很多问题其实不是在网络上啊。“数据库读写太多”这应该就不用举例子了，典型的 N+1 问题就是，尤其出现在 ORM 框架上。由于国内倾向于使用 MyBatis，所以这样的问题不多见，但我个人来说其实有时也能看到真有人拿 MyBatis 写出 N+1 问题的。我见到有一些实现得很糟糕的接口其实根本不需要分页，实际上根本就是拿分页掩盖了背后过多的查询语句，这种就属于完全不必要分页的

“后端计算量大”看起来是个没辙的问题，其实

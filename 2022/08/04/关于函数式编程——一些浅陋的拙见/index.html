关于函数式编程——一些浅陋的拙见

关于函数式编程——一些浅陋的拙见

如果有读者是被头图吸引进来的，可能要失望了，因为本文既不讲 λ 演算，也不讲 Elixir。只是谈一些浅陋的拙见。

正如标题所示，这不是一篇专业的文章，只是我个人关于函数式编程产生的一些杂乱无章（而且很可能不正确）的想法。我不是个专业的函数式编程程序员。实话实说，函数式编程技巧对我而言主要使用在 JavaScript、Java 和 Python 中，我从没有真的实际拿纯函数式语言开发过什么像样的应用。

我对函数式编程的了解主要来自几本小薄书和网络上碎片化知识的获取。对于那些偏向函数式的语言，我基本只有 LISP 语系（Scheme 与 Clojure）和 Scala 是相对熟悉的，但也并没有真的拿它们开发过什么超越玩具性质的项目。我离散数学学得很差，对 PL 的认知也很浅，甚至现在要让我背诵一下群论中的某些基础定义，我一时半会也想不起来。我对 Haskell 和 ML 语系也只有一些比较浅薄的了解，因此这一定是一篇很不专业的文章。

我知道一定有富有经验的读者读到这里就意识到继续读本文是浪费时间，然后划走本文了，我很尊重这些知识水平超过我的读者，也希望这些读者真的不要在本文上浪费时间了。但是，如果你是和曾经的我一样对函数式编程几乎一无所知的新手，或许我这些冗长的碎碎念能给你一些启发。

函数式编程是未来吗

这要看如何定义“未来”。如果“未来”指基础设施开发，那么我不排除函数式编程成为“未来”的可能，毕竟在分布式越来越火的当下，函数式编程以很优雅的方式解决了并发问题，且具有很高的安全性与较快的开发速度（对于熟悉者而言），相比之下它在学习成本和编译速度上的缺点就不是那么严重了，毕竟开发基础设施的团队往往具有较高的素质，这对他们来说不会是很大的问题。

如果“未来”指底层硬件开发，那别想了，底层开发要求的是尽可能压榨硬件性能。在性能极度敏感的情况下，如操作系统编程，面向对象都嫌复杂，函数式编程编译出来的代码再优化效率也比不过纯过程式，肯定取代不了。如果要问有哪门编程语言三十年后还能活着，其他语言我都不敢百分百肯定，但 C 语言肯定能活着，还能活得好好的。

如果“未来”指业务开发，很遗憾，我不认为函数式编程能翻起太大的浪花，至少一定无法“替代”面向对象编程。即使是 Scala 这种把 OOP 和 FP 糅在一起，且倾向于 FP 的语言也做不到。面向对象应当仍然会作为项目代码的主体，然后辅以少量的函数式代码实现逻辑细节，如经典的列表数据处理。至于为什么，我将在下面谈函数式编程的问题时指出。

所以说有几点可以总结出来：

  • 摩尔定律的失效逐渐使机器性能的增长速度渐渐无法满足高速增长的性能需求，因此多核与分布式方案得到了愈发广泛的应用，这使得并发编程受到重视，而在这方面，毫无疑问函数式编程的思路比起面向对象的思路相性更好。
  • 得益于基础工作的完善，如今很多任务已经不需要程序员从头开始用命令式的逻辑去写执行步骤了。在业务逻辑相对固定的情况下，大多数业务实现都可以成为一连串函数的组合，因此函数式编程由于其更好的可读性与简洁性受到欢迎。
  • 对于底层硬件开发，函数式编程由于性能问题很难真正被大规模使用。
  • 对于更高层级的业务逻辑组织，函数式编程由于其固有的复杂性不太可能成为主流。纯 FP 想在这一层面上发挥作用不大可能写出比 OOP 更简洁易读的代码，不如说反而可能会更加冗长难读。
  • 由于 GUI 开发不再是主流，这一与 OOP 最契合的使用场景之一也渐渐没落了，那么函数式编程自然也少了一些劣势。
  • 显然，纯函数式编程几乎不可能在实际项目成为主流。或者说，至少是多人团队开发的项目中不太可能使用纯函数式编程，这带来了过重的思维负担。如 WhatsApp 当初五十个人使用 Erlang 开发后端已经是极限了。

那么，有没有可能函数式编程会成为真正的“未来”呢？有可能，还不少，可惜都不太现实：

  • 发生核战争或致死率极高的传染病，人类死了一大半，大家自然就不用怎么考虑超大规模业务了，少人团队甚至单人开发都能成为常态，那自然可以怎么爽怎么来。
  • 计算机行业因不明原因受到重击，开发者大规模减少，凑巧的是这些留下来的开发者水平都很不错。这时使用超大型团队维护超大规模项目的逻辑已经行不通了，这时若仍需维持之前的业务规模，就必须要找到能够充分发挥个体生产力的编程方式，那么函数式编程显然会被主流采用。而由于遗留下来的开发者水平都很不错，因此函数式编程学习成本过高的问题也就不是问题了。

但是，也有更多的可能会使函数式编程比现在更凉，而且这些可能甚至更贴近现实：

  • 物理学迎来又一次突破，摩尔定律再次生效，单核性能飞速上涨，并行编程热度大幅下降，函数式编程的“无副作用”、“不变性”变得不再那么具有吸引力。
  • 机器性能在横向发展上也遇到了瓶颈。由于性能发展长时间停滞，函数式编程的“不变性”思路反而成为了性能负担，有状态的“对象”更受青睐。

可以看到，FP 受到的限制很多，几乎只在特定时代背景下具有优势。比如机器性能问题既可以成为函数式编程逐渐受到欢迎的原因（摩尔定律失效，性能提升开始追求多核与分布式，即横向扩展），也可以成为函数式编程再一次没落的理由（机器性能长期停滞，函数式编程遇到性能瓶颈）。

函数式编程的问题

我认为很难是。函数式编程最大的问题在于其学习成本太高，且对使用者有较高的数学要求。十多年前就有人预言函数式编程（FP）会逐渐代替面向对象编程（OOP），然后十几年过去了，并没见太大起色。说到底，这几年函数式编程有些抬头的兆头，其实要感谢 JavaScript（和更早的 Java 8）。我知道，很多 Haskell 使用者瞧不起 JS，但现在在 Google Images 上搜索 Functional Programming，一眼看下去大部分都是 JS 代码。这其实有点讽刺，纯函数式语言发展了这么多年，指望出一个能大规模被应用于工程的 FP 语言，结果 FP 最近渐渐有起色居然要归功于 JavaScript——一个因早年设计缺陷而只能拥有残废 class，因而不得不诉求于 FP 的语言。

老实说，我认为目前前端用的那点充其量只能算是“技巧”的“函数式编程”已经接近函数式编程在业务逻辑中的使用极限了。当然，我不排除函数式编程逐渐在基础设施开发中越来越受欢迎的可能，毕竟函数式编程能够以非常巧妙和优雅的方式解决并发问题，不如说我很看好函数式编程在这方面的发展。

可以看到，2014 出现的 Java 8 中引入的 lambda 和 stream 如今都未被完全接受，反倒是前端这边因为 React 的偶然崛起以惊人地速度接受了函数式编程的一些思想。很多人会指出，函数式编程的一些核心思想如类型系统、柯里化、模式匹配等思想无论在前端还是后端开发中都未被广泛使用，将这些浅薄的应用称为“函数式编程”并不合适。但是，即使是这些“浅薄”的应用推广起来也困难重重。

让我们思考程序员为什么从一种编程范式切换到另一种编程范式：

  • 更简洁可读的代码
  • 更好地组织代码结构
  • 减少出错的可能（更高的安全性）
  • 更强的可维护性
  • 减少程序员的思维负担
  • 更高的开发效率
  • ……

上面只列举了少数几点。观察历史，从滥用goto到结构化编程，从结构化编程到面向对象编程，都符合上面这几点。有趣的是，这几点中（也包括未列出来的更多点）并未包括“更高的性能”。毫无疑问，理论性能最强的是手写汇编，接着是goto乱飞把高级编程语言当汇编使的方式，但这两种方式显然都会为代码带来极高的维护成本，而且这种“高性能”也只存在理论上——很少有人的脑子能在这种编程范式下塞下

声明式与命令式

面向对象（OOP）与函数式编程（FP）之争，其实本质在于命令式与声明式编程之争。我们有时会说，“OOP 与 FP 并不冲突”，我也会说这样的话。但老实说，纯 FP 的代码几乎是掺不进一点 OOP 的代码的。OOP 核心在于状态和消息，FP 核心在于函数的组合与不变性。函数没有状态，因此根本上是与存在状态的 OOP 对象冲突的。
